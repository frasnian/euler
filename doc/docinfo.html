Euler Zero - About the local docs & customizing options

Each solution has its own 'readme' and 'detail' files that are located in the directory for that solution. These files contain the following information:

"Readme" files:
The "readme" files are intended to be your friend. They can generate build command lines for each solution/variant in the repository. These files contain a brief summary of the original problem from Project Euler (problems that do not have a simple text description or require graphic elements will not be included in readme files beyond a summary - the full problem will be in the detail file). They will include any relevant build notes, and as noted, can generate build command lines for compilers supported by that solution. The readme file will also have the C++ Standard and minimum compiler versions required for solution variants.

The default compiler and build options used for command lines generated the "readme" files are documented here, and can be customized with a .euler‑zero‑prefs.js file (see below).

"Detail" files:
The "detail" files contain any implementation notes and the approach taken for that solution, as well as the full problem description from Project Euler. They are basically the "WTF were you thinking?" pages. Any differences between calculation-at-execution and zero-runtime variants are documented in this file, and (possibly) more detailed descriptions and notes about the source code beyond that provided in the source file comments.

The .euler-zero-prefs.js file
The default build options used by the readmeNNN.html for each solution to generate command lines can be overridden with a .euler‑zero‑prefs.js file. This file is listed in the .gitignore for the repo, so go nuts. A default‑euler‑zero‑prefs.js file has commentary and details about what you can change here. The default prefs file isn't included by any doc files - it's only intended as documentation for what you can customize and as a template for your own local .euler‑zero‑prefs.js file.

You can also change the default page loaded in the main content frame from index.html (example: you may want to set it to the Solution Index page - after reading the other docs, of course :), various options such as Compiler Explorer (see below), etc.

Doc colors, formatting, etc.
Most of this can tweaked with euler‑zero.css or common.js. The javascript may be ugly in some places, but it's simple and serviceable. I did not want to have dependencies on external things like Node.js and -again- this is not a serious project, so a quick-and-dirty approach is sometimes used.

Compiler Explorer option
Compiler Explorer is an online tool developed by Matt Godbolt, and it absolutely rocks. If you want to test code against various compilers and compiler versions that you might not have installed, it's an ideal solution to check compiler compatibility and assembly output for your code. It also has a code editing pane, so it's not cut-and-paste only. I use this thing all the time, and in all likelihood you probably will to (he recently created a Patreon page for this project - so please chip in to help fund it or buy him a beer or something. He deserves it for providing this service for years).

As noted on the Build Info page, it's actually much easier to use Compiler Explorer if all you want is the generated assembly - paste in the appropriate compiler options from the solution readme file, paste in the source code for the variant you want into the input window, choose a supported compiler and voila! This has the added benefit of eliding extraneous section info (e.g. comment,ident,note, etc.) so the output is a lot cleaner.

I use this option pretty much all of the time, so you can enable it by default in .euler‑zero‑prefs.js. Setting this option causes the readme to blow off options like input & output filenames and "generate assembly" when generating a command line, as they are already implied. Note that online compilation environments obviously cannot handle local include files, so to accommodate this restriction some solutions have source files with a lot of data that would normally be included in a separate header file.
